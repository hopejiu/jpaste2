## 修复 JSON/图片/curl/ws 查看器窗口白屏卡死

**根因**：前端路由用 wouter + `useHashLocation`（hash 路由），但 `src-tauri/src/command/viewer.rs` 的四个 `open_*_viewer` 命令用 path 形式 URL `WebviewUrl::App("/viewer/json?id={id}")`。hash 为空 → wouter 匹配不到路由 → 白屏；查看器页 `URLSearchParams(window.location.hash.split('?')[1])` 读不到 id。

**修复**：四处 URL 改为 hash 形式 `#/viewer/json?id={id}`（与已正常工作的 toast 窗口 `#/toast?...` 模式一致）。actions/*.ts 的 fallback `window.location.hash = '/viewer/...'` 无需改（浏览器赋值 hash 时自动补 `#`）。

## 修复 toast 死锁 + 日志改本地时间

**toast 死锁根因**（症状：通知开+复制→整个 app 冻死，快捷键/托盘全失灵；通知关→正常）：剪贴板事件在后台线程处理，整个处理过程持有 `state.lock()`，期间调用 `create_toast_window_inner`→`WebviewWindowBuilder::build()`。从非主线程建窗会派发到主线程并阻塞等待；新 toast 窗口抢焦点→主窗口 `Focused(false)` 事件处理器（on_window_event）尝试 `state.lock()`→后台线程持锁等主线程建窗，主线程等锁→死锁。

**修复**：`src-tauri/src/lib.rs` 的 `on_clipboard_change` 中，将 toast 参数收集到 `Option<(String,f64)>`，在块作用域结束时释放 state 锁，再创建 toast 窗口。加 `ponytail:` 注释说明此约束（所有此线程建窗路径必须先释放 state 锁）。

**日志本地时间**：`log_service.rs` 的 `chrono_now()` 原为手写 UTC 计算（含 is_leap/days 算法），改为 `chrono::Local::now().format("%Y-%m-%d %H:%M:%S%.3f")`。文件名 pattern 改用 `chrono::Local::now().format("%Y-%m-%d-%H")`。chrono 已在 Tauri 间接依赖树中，加为直接依赖不增体积。删除 is_leap 及对应测试。

**诊断日志**：`viewer.rs` open_json_viewer 加 `building window url=` / `build OK` / `build FAILED`；`lib.rs` create_toast_window_inner 加 `creating new window url=`。

**注意**：`tauri dev` 下前端改动 HMR 生效，但 Rust 改动需重启 dev 进程才重新编译。之前日志看不到 Rust 新日志就是因为没重启。

## 修复 Tauri WebviewUrl::App 不支持 hash URL + invoke 错误吞咽

**根因**：`WebviewUrl::App("#/toast?...")` / `WebviewUrl::App("#/viewer/json?id=2")` 在 Tauri v2.11.5 下无法正确加载 index.html——Tauri 把 `#...` 当作 path 段处理，webview 收不到 SPA 入口页 → 前端 main.tsx 从不执行（日志无 `jPaste v2 starting...`）→ toast 404 / json 白屏。主窗口正常是因为它走 tauri.conf.json 的 devUrl，不经 WebviewUrl::App。

**修复**：所有新窗口改用 `WebviewUrl::App("/".into())` 加载 SPA 根，再用 `.initialization_script()` 在 DOMContentLoaded 时 `window.location.hash = '/viewer/json?id=2'`（不带前导 #，浏览器赋值 location.hash 自动加 #）。
- `src-tauri/src/command/viewer.rs`：提取 `build_viewer_window` helper 统一处理 4 个 viewer
- `src-tauri/src/lib.rs`：toast 新建窗口同样改用 `/` + init_script；复用窗口的 eval 设 hash 保持（用不带 # 的值）

**invoke 错误吞咽修复**：`src/actions/json.ts` 原 `.catch(() => fallback)` 吞掉了 invoke 错误，导致 Rust 端日志缺失时无法定位。改为 `.catch((e) => { error('invoke FAILED', e); fallback })`，让 Tauri 命令调用失败可见。

## initialization_script 时机修复

`DOMContentLoaded` listener 时机不可靠（toast 显示主页而非 toast 页，说明 hash 没设）。改为：init_script 存 `window.__INITIAL_HASH__`，main.tsx 启动时读取并 `location.hash = __INITIAL_HASH__`，确保在 wouter 挂载前生效。加 build 前后日志确认是否阻塞。

## build() 阻塞修复 + 日志文件缺失修复

**build 阻塞根因**：日志确认 `build_viewer_window: calling build()...` 后无任何输出（无 build OK 无 FAILED）→ `WebviewWindowBuilder::build()` 从 Tauri 命令线程同步调用时阻塞，永不返回。build 会派发到主线程并等，但命令线程的同步阻塞阻止主线程执行创建任务 → 死等。

**修复**：`viewer.rs` 四个 `open_*_viewer` 改为 `async`，`build_viewer_window` 用 `tauri::async_runtime::spawn_blocking` 在 blocking 线程执行 build 并 await。async 命令在 Tauri runtime 上运行，yield 后主线程能处理窗口创建。

**日志文件缺失根因**：`log_service.rs` 的 `init_global_logger` 给 LogBridge 用 `svc.try_clone()`，而 try_clone 原实现返回 `writer.file=None` 的新 LogService → `log::info!` 等宏走 LogBridge 时只写 stderr 不写文件。前端 `frontend_log` 走原始 svc（有 file）→ 写文件。所以 Rust 日志只在终端，前端日志在文件+终端。

**修复**：LogService 的 `writer` 字段从 `Mutex<LogWriter>` 改为 `Arc<Mutex<LogWriter>>`，`try_clone` 用 `Arc::clone` 共享同一 writer。LogBridge 和主 svc 现在写同一文件。

**待验证**：toast 创建路径（`create_toast_window_inner`，后台线程调用）可能也需要 spawn_blocking，且 init_script 方案对 toast 是否生效待确认。

## 第三波修复：日志确认 External URL 可以正常加载，改用 window.location.href 整页导航替代 hash

## 第四波修复：wouter v3 matchRoute 不剥离 query string + toast 整页导航导致 WebView2 崩溃

**关键突破**：用户打开 JSON viewer 后 404 页面显示 `hash: #/viewer/json?id=22`，URL 和 hash 完全正确，说明整页导航方案生效。但 wouter 仍然不匹配路由。

**根因 1：wouter v3.10.0 matchRoute 不剥离 ?query**：`useHashLocation` 返回 `/viewer/json?id=22`，`regexparam` exact-match regex `/^\/viewer\/json$/` 遇到 `?id=22` 失败。修复：新建 `src/hooks/use-hash-path-location.ts` 包装 `useHashLocation`，在传给 Router 前 `split('?')[0]`。

**根因 2：toast 整页导航导致 reuse eval 崩溃**：dev 模式 init_script 中 `window.location.href = 'http://localhost:1420/#/toast?...'` 触发整页导航后，webview 处于过渡态。第二次 toast 尝试 `window.eval("location.hash=...")` 失败 WebView2 `HRESULT(0x8007139F)`。修复：toast 窗口创建和复用**全部改用 `window.location.hash = '...'`，禁止整页导航**。Toast 的 SPA 在 `WebviewUrl::App("/")` 上启动，init_script 和 eval 仅设 hash。

**改动文件**：`src/hooks/use-hash-path-location.ts`（新建）+ `src/app.tsx`（改用新 hook）+ `src-tauri/src/lib.rs`（toast 简化）
